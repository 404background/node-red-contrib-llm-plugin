# Vibe Schema — intermediate flow format

Vibe Schema is the alias-based JSON format the LLM produces instead of raw
Node-RED JSON. Conversion in both directions is implemented by
`flow_converter_core.js`.

## Why an intermediate format?

A raw Node-RED flow has three properties that hurt LLM accuracy:

1. **Random IDs** — every node has a 16-character random `id` that the LLM
   has no way to invent meaningfully. References (between wires, config
   nodes, group memberships, ...) all hinge on those IDs being globally
   unique.
2. **Pixel coordinates** — `x` / `y` must be set on every canvas node, but
   the LLM has no reliable model of the canvas viewport, node widths, or
   chain spacing.
3. **Nested wire arrays** — `wires: [[]]` is one wire array per output
   port; each entry is itself an array of target IDs. Easy to malform.

Vibe Schema replaces all three with:

1. **Human-readable aliases** — keys like `inject_trigger`, `function_format`.
   Globally unique within the schema, derived from `{type}_{name}`.
2. **No coordinates** — generated by the layout engine (see
   [LAYOUT.md](./LAYOUT.md)) when the schema is converted back.
3. **Flat connection list** — `connections: [{ from, to, fromPort? }, …]`.

## Original Node-RED JSON (for reference)

This is what Node-RED exports when you copy a flow or call
`RED.nodes.createExportableNodeSet`. Every canvas node carries an
absolute pixel `x` / `y`, a workspace reference `z`, and a nested
`wires` array. Config nodes (`mqtt-broker` here) live outside the
canvas — no `x` / `y` / `wires`, just configuration properties.

```json
[
  {
    "id": "abc123",
    "type": "tab",
    "label": "Sensors",
    "disabled": false,
    "info": ""
  },
  {
    "id": "n1",
    "type": "inject",
    "z": "abc123",
    "name": "Tick",
    "props": [{ "p": "payload" }],
    "repeat": "5",
    "crontab": "",
    "once": false,
    "payload": "",
    "payloadType": "date",
    "x": 100,
    "y": 80,
    "wires": [["n2"]]
  },
  {
    "id": "n2",
    "type": "function",
    "z": "abc123",
    "name": "Format",
    "func": "msg.payload = { ts: msg.payload };\nreturn msg;",
    "outputs": 1,
    "noerr": 0,
    "x": 280,
    "y": 80,
    "wires": [["n3"]]
  },
  {
    "id": "n3",
    "type": "mqtt out",
    "z": "abc123",
    "name": "Publish",
    "topic": "sensors/ticks",
    "qos": "0",
    "retain": "false",
    "broker": "broker_cfg",
    "x": 460,
    "y": 80,
    "wires": []
  },
  {
    "id": "broker_cfg",
    "type": "mqtt-broker",
    "name": "Local broker",
    "broker": "localhost",
    "port": "1883",
    "clientid": "",
    "usetls": false,
    "keepalive": "60"
  }
]
```

What the LLM has to invent here, every time:

- `id` for every node — 6-to-16-character random strings, all unique.
- `x` / `y` for every canvas node — pixel coordinates that don't
  overlap and visually flow left-to-right.
- The `wires: [[targetId]]` array of arrays per output port, with
  exactly the right `targetId` referencing another generated `id`.
- The `broker: "broker_cfg"` reference in `mqtt out` — must exactly
  match the `id` of an actually-present `mqtt-broker` config node.

Getting any of these subtly wrong produces a flow that won't import,
or imports broken. Vibe Schema removes all four problems below.

## Schema shape

```json
{
  "description": "string (optional, auto-generated)",
  "applyMode": "merge | edit-only | overwrite | delete-only",
  "nodes": {
    "<alias>": {
      "type": "<node-red-type>",
      "name": "<optional display name>",
      "flow": "<tab label, canvas nodes only>",
      "config": true,
      "props": { "<type-specific>": "<value>" }
    }
  },
  "connections": [
    { "from": "<alias>", "to": "<alias>", "fromPort": 0 },
    { "remove": { "from": "<alias>", "to": "<alias>" } }
  ]
}
```

### Node entry fields

| Field | Type | Purpose |
|-------|------|---------|
| `type` | string | Required. Node-RED node type (`inject`, `function`, `mqtt-broker`, …). |
| `name` | string | Display name. Used for the on-canvas label and alias generation. |
| `flow` | string | Canvas nodes: tab label this node belongs to. Required in multi-flow context; defaults to the ACTIVE FLOW. |
| `config` | bool | Set to `true` for config nodes (they live outside the canvas). |
| `props` | object | Type-specific properties. Anything not in the META keys (`id`, `type`, `name`, `z`, `x`, `y`, `wires`, `g`). |

Any root key besides `type`, `name`, `flow`, `config`, `props`,
`_llmAlias`, `_llmFlow` is treated as a property and flattened on
conversion — so either nesting style works:

```json
{ "comment_x": { "type": "comment", "name": "Title", "props": { "info": "Body" } } }
{ "comment_x": { "type": "comment", "name": "Title", "info": "Body" } }
```

### Alias generation rules (`generateAlias`)

`{type}_{name}` with these normalizations:
- `[^a-zA-Z0-9_]` → `_`
- collapse `_{2,}` → `_`
- trim leading / trailing `_`
- lower-case
- empty result → `node`

Collisions get a numeric suffix: `inject_trigger`, `inject_trigger_2`, …

### Map-to-null deletion

To delete an existing node, set its alias value to `null`:

```json
{ "nodes": { "inject_to_remove": null } }
```

### Connection delete directive

To remove a specific edge while keeping both endpoints:

```json
{ "connections": [ { "remove": { "from": "inject_a", "to": "function_b" } } ] }
```

### Apply mode

Top-level `applyMode` field decides how the importer merges the schema
into the existing flow:

| Mode | Effect |
|------|--------|
| `merge` | Default. Add new nodes, update matched existing ones, leave others alone. |
| `edit-only` | Only update existing nodes; new nodes are blocked. |
| `overwrite` | Replace the entire workspace with the schema (no preservation). |
| `delete-only` | Apply only the deletion directives; ignore additions. |

## `toIntermediate(nodeRedJson, options?)` — Node-RED → Vibe Schema

```js
const { toIntermediate } = require('./flow_converter_core.js');
const schema = toIntermediate(exportedNodes, { includeIdMap: true });
```

Pipeline:

1. **Filter** — drop `tab` and `subflow:*` definition nodes.
2. **Pass 1: assign aliases** — `generateAlias(node, used)` for every node;
   `idToAlias[node.id] = alias`.
3. **Pass 2: build entries** — for each node:
   - Collect META-keys-stripped properties into `props` (skip
     editor-internals starting with `_`).
   - For `inject`, drop the internal `props` array (toNodeRed regenerates it).
   - Resolve string props that match a known node ID → replace with the
     corresponding alias (so the intermediate format is portable).
   - Set `config: true` if `isConfigNode(node)`.
4. **Build connections** — for each node's `wires[port][i] = targetId`,
   look up `idToAlias[targetId]` and emit `{ from, to, fromPort? }`.
5. **Description** — auto-generate a short type summary
   (`"3 node(s): inject, function, debug"`).
6. **Optional id-map** — `options.includeIdMap = true` attaches
   `_meta.idToAlias` (used by the importer for multi-flow grouping).

The schema is round-trippable: `toNodeRed(toIntermediate(flow))` produces a
functionally equivalent flow (fresh IDs and laid-out coordinates).

## `toNodeRed(intermediate, options?)` — Vibe Schema → Node-RED

```js
const nodes = toNodeRed(schema, { workspace: 'tabId', startX: 100, startY: 100 });
```

Pipeline:

1. **Auto-stub missing config refs** — if a prop value looks like an alias
   but doesn't appear in `schema.nodes`, create a stub config node for it:
   - prop key ends in `config` (e.g. `venvconfig`) → type
     `<key>-config`; or
   - prop key is a known reference (`broker`, `group`, `tab`, `base`,
     `serialport`) → map to its config type.
   Stubs are marked `_autoStub: true` so the importer doesn't overwrite
   real existing config nodes with empty props.
2. **Generate real IDs** — fresh 11-char random ID per alias.
3. **Separate canvas vs. config aliases** — config nodes (`isConfigType` or
   `config: true`) skip coordinate / layout assignment.
4. **Build adjacency** — `outgoing` / `incoming` from `connections`,
   skipping edges whose target is a `NO_INPUT_TYPES` member (the wire
   would render but never carry messages) or whose source is a
   `NO_OUTPUT_TYPES` member (e.g. `comment`).
5. **Layout** — delegate to `CanvasLayout.layoutNodes` for the logical
   `{col, row, comp}` positions, then `CanvasLayout.computeComponentYOffsets`
   for the per-component y offset. Horizontal placement is width-aware
   (see [LAYOUT.md](./LAYOUT.md)): each column's x-centre is derived from
   the previous column's right edge plus `edgeGap`, so wide-named nodes
   automatically push their column over without overlapping the next one.
6. **Build wires** — convert `connections` to per-port arrays
   `wires[fromPort].push(toId)`, dropping invalid endpoints as in step 4.
7. **Assemble nodes** — for each alias:
   - Carry `id`, `type`, optional `name`.
   - Optional `_llmAlias` (when `preserveAlias: true`) so the importer can
     fuzzy-match against existing nodes.
   - Optional `_llmFlow` from `spec.flow` for multi-flow routing.
   - Set `z` to the workspace (canvas nodes only).
   - Flatten `props` and root-level keys (skipping META + Vibe-specific
     keys: `props`, `_llmAlias`, `config`, `flow`).
   - Resolve alias references in flattened props (`broker: "my_mqtt"` →
     real ID).
   - Set `wires` from the connection map (canvas only).
8. **Type-specific normalisers** — `inject` (regenerate `props` rules),
   `function` (require / libs extraction, format code, default outputs),
   `change` / `switch` (rules), `template` (defaults).

## Detection helpers

Exported from `flow_converter_core.js` alongside the converters:

| Helper | What it returns |
|--------|-----------------|
| `isVibeSchema(obj)` | `true` iff `obj.nodes` is a plain object and `obj.connections` is an array. |
| `isConfigType(type)` | Runtime: `RED.nodes.getType(type).category === 'config'`. Static fallback: type ends in `-config`. |
| `isConfigNode(node)` | Type-based + structural (no `x`/`y`/`wires`/`g`). Excludes `tab` and `subflow:*`. |
| `isCanvasNode(node)` | `!tab && !subflow: && !isConfigNode(node)`. |
| `isNoInputType(type)` | Runtime `inputs === 0`, or static list (`inject`, `catch`, `status`, `complete`, `http in`, `mqtt in`, `websocket in`, `tcp in`, `udp in`, `comment`). |
| `isNoOutputType(type)` | Runtime `outputs === 0`, or static `comment` only. |
| `setRuntimeGetType(fn)` | Inject `RED.nodes.getType` so the helpers above see community / custom node defs. |

## Configurator alias

For backward compatibility, the browser global exposes the API under both
names:

```js
window.LLMPlugin.FlowConverterCore === window.LLMPlugin.Configurator
```

New code should prefer `FlowConverterCore`.

## Round-trip example

Original Node-RED export:

```json
[
  { "id": "abc123", "type": "tab", "label": "Sensors" },
  { "id": "n1", "type": "inject",   "name": "Tick", "z": "abc123", "x": 100, "y": 80, "wires": [["n2"]], "payload": "", "payloadType": "date" },
  { "id": "n2", "type": "function", "name": "Fmt",  "z": "abc123", "x": 280, "y": 80, "wires": [["n3"]], "func": "return msg;", "outputs": 1 },
  { "id": "n3", "type": "debug",    "name": "Out",  "z": "abc123", "x": 460, "y": 80, "wires": [], "active": true }
]
```

`toIntermediate` →

```json
{
  "description": "3 node(s): inject, function, debug",
  "nodes": {
    "inject_tick": { "type": "inject", "name": "Tick", "props": { "payload": "", "payloadType": "date" } },
    "function_fmt": { "type": "function", "name": "Fmt", "props": { "func": "return msg;", "outputs": 1 } },
    "debug_out": { "type": "debug", "name": "Out", "props": { "active": true } }
  },
  "connections": [
    { "from": "inject_tick", "to": "function_fmt" },
    { "from": "function_fmt", "to": "debug_out" }
  ]
}
```

`toNodeRed` (with `workspace: 'abc123', startX: 100, startY: 80`) →
equivalent flow with fresh IDs and recomputed `x` / `y` via the layout
engine.
